# M16 — SEO-Safe Site Migration (design)

**Date:** 2026-07-02
**Status:** awaiting owner review
**Milestone:** M16 (blocks the apex cutover — see `docs/go-live-checklist.md` §3)
**Parent plan:** `docs/seo-migration-plan.md` (analysis + P0–P2 findings)

## Why

Replacing the WordPress site on `ceylonhop.com` with the new static site deletes the per-route
landing pages that carry the long-tail rankings (*"Kandy to Ella shared taxi"* etc.). M16
recreates that route-page layer in the new site, 301-maps every old URL, and ships the missing
SEO plumbing (sitemap, robots, 404, canonicals, structured data) so the cutover is
ranking-neutral or better.

## 1. Route pages

**Generator:** `tools/generate-route-pages.mjs` (Node, no deps). Loads `transfers-data.js` with a
`window` shim → `TRANSFERS` gives real km, duration, engine-parity car/van prices
(`privateQuote`) and shared-seat availability/price (`sharedOption`). Merges with hand-written
copy in `tools/route-content.json` (per-place blurbs + per-pair route narrative, direction-aware
intros). Emits `trip/<from>-to-<to>/index.html` + `trip/index.html` + `sitemap.xml` + redirect
stubs + the Cloudflare CSV. Run via `npm run generate` (root `package.json` script); a test
regenerates and diffs so committed output can't drift from the data (codegen-check pattern).

**The 22 pairs** (each generating both directions):
airport↔kandy · airport↔sigiriya · airport↔galle · airport↔mirissa · airport↔ella ·
airport↔negombo · airport↔colombo · negombo↔sigiriya · negombo↔kandy · colombo↔kandy ·
colombo↔galle · colombo↔ella · sigiriya↔kandy · kandy↔ella · kandy↔nuwara-eliya ·
nuwara-eliya↔ella · ella↔yala · ella↔arugam-bay · ella↔mirissa · yala↔mirissa ·
mirissa↔galle · galle↔ella

**Page anatomy** (matches site design: `site.css`, header/footer cloned from existing pages,
relative asset paths `../../` so pages work on the github.io subpath today and the apex later):
- Unique `<title>` / meta description / self-canonical (absolute `https://ceylonhop.com/trip/…/`).
- H1 (*"Kandy to Ella — private transfer & shared ride"*), price-from block (car / van / shared
  seat where the corridor has one), real km + drive time.
- Direction-aware route narrative + what-you-pass highlights; train/scenic notes where relevant
  (hand-written content, not template filler).
- 4–5 FAQ entries (duration, cost, luggage, stops, booking) — auto-filled numbers.
- At least one real `<img alt>` reusing existing site imagery where a suitable image exists.
- CTAs: `search.html?from=<id>&to=<id>` (prefilled search) + WhatsApp.
- JSON-LD: `BreadcrumbList`, `FAQPage`, `Service` (+`Offer`, USD price-from). No aggregateRating.

**`/trip/` index:** a routes directory page listing all 44, grouped by hub — redirect target for
old `/routes/` + `/trip/`, and the internal-linking spine (each route page breadcrumbs to it).

## 3. Site plumbing

- **`sitemap.xml`** — generated: the 44 route pages, `/trip/`, index/about/why/blog/tours/plan,
  terms, privacy. Absolute apex URLs. (Inert until cutover; the sitemap is submitted in GSC then.)
- **`robots.txt`** — allow all + `Sitemap: https://ceylonhop.com/sitemap.xml`. (Project-page
  subpath means it only takes effect once on the apex.)
- **`404.html`** — branded, links to `/trip/` + home (GitHub Pages serves it natively).
- **Head fixes on the 8 existing pages** (sanctioned unfreeze, S4): self-canonical on every page;
  `noindex` on `search.html` + `booking.html` (param-driven); OG tags sitewide; index.html
  structured data → truthful `TravelAgency` (telephone, areaServed, `sameAs` → TripAdvisor
  listing) without aggregateRating; rating copy corrected per S2.
- **No CNAME yet** (S8). Canonicals pointing at the apex from github.io are safe — they
  de-emphasize the dev site rather than promote it.

## 4. Testing (web-tests, existing Vitest + Playwright harness)

- **Codegen check:** regenerate → `git diff --exit-code` on generated files (prices/distances in
  pages always match `transfers-data.js`).
- **Static invariants (Vitest over fs):** unique title/desc/H1/canonical per page; sitemap ↔
  files 1:1; every redirect stub target exists; every route page's relative asset refs resolve;
  no page besides search/booking carries `noindex`; no `aggregateRating` anywhere.
- **Playwright:** a route page renders with header/nav/prices; its CTA lands on `search.html`
  with from/to prefilled; a redirect stub navigates to its target.

## Implementation addenda (discovered during scouting, 2026-07-02)

These are mechanism-level findings that refine *how* the above is built; they do not change the
approved scope.

- **A1 — Freeze model must be refined first (new Step 0).** The local PreToolUse hook
  (`.claude/hooks/protect-ui.sh`) blocks **every** `*.html` write *by basename*, so it would block
  the new route pages, redirect stubs, and `terms/privacy/404.html` — not just the existing frozen
  files. The CI gate (`.github/workflows/ci.yml`) is narrower — it only matches **root-level**
  html (`^[^/]+\.html$`). Step 0 converts both guards from a blanket `*.html` rule to an **explicit
  list of the existing live-site files** (the 10 current root html — `index, about, blog, booking,
  plan, search, tour, tours, why, _ops-preview` — plus `site.css`, `favicon.svg`,
  `image-slots.state.json`, and the frozen `*.js`). Every currently-protected file stays protected;
  genuinely-new SEO files are permitted. This is a `.claude/` + `.github/` change (both allowed).
  PR3's edits to the existing pages still require the labelled unfreeze.
- **A2 — Route pages carry a self-contained header/footer.** `site.js` injects the nav/footer into
  `data-header`/`data-footer` using **page-relative** hrefs (`href="index.html"`), which break from
  `/trip/<slug>/`. Route pages therefore ship a **static header/footer emitted by the generator with
  `../../`-relative links** (robust on the apex, github.io project path, and local file serving), plus
  a tiny inline mobile-menu toggle. They still load `site.css` (`../../site.css`) so styling matches.
  The generator owns this markup; regenerating re-syncs it.
- **A3 — Real `<img>` is deferred (owner content dependency).** The site has no image files on disk;
  imagery is data-URIs in the frozen `image-slots.state.json`, of which **only `hero-photo` is
  filled**. Route pages ship with the site's CSS-gradient hero (no broken drop-zones, no 44× repeated
  base64 blob). The P2 "real `<img alt>`" SEO item needs owner-supplied route photos — flagged as a
  founder input; pages leave a documented hero area to slot them in later.
- **A4 — Generator loads `transfers-data.js` via a `window` shim.** The file is a browser IIFE that
  only assigns `window.TRANSFERS` (no DOM). The Node generator sets a `window` global, evaluates the
  file in a sandbox, and reads `TRANSFERS` — the single source of prices/distances, so generated
  pages can't diverge from the engine-parity rate card.
